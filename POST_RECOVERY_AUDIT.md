# Post-Recovery Audit

## Executive summary

Deployment recovery is functionally successful for the core service plane:

- Render build/startup is healthy per the provided deployment context.
- The active FastAPI entrypoint is `app.main:app` in `app/main.py`.
- `GET /health` is implemented and returns `200` in code (`app/main.py:133-135`).
- Vertex credentials are loaded from `GCP_KEY_B64` via `app/services/vertex_imagen.py:_ensure_credentials_from_b64`.
- Poster generation is succeeding end-to-end and uploading artifacts to R2 via `app/services/storage_bridge.py:store_image_and_url` and `app/services/r2_client.py`.

The remaining production issue is not general deployment health. It is a degraded runtime branch in the kitposter pipeline:

- `POST /api/generate-poster` is healthy enough to return `success`.
- The poster artifact is stored and `poster_url` is returned.
- But the kitposter edit path is disabled before the edit call is attempted, so the service falls back to locked-frame/local template rendering and marks the response `degraded=true`.

In short: deployment is healthy; output quality is degraded because the intended edit pipeline is currently unavailable.

## What is already healthy

- FastAPI boot path: `uvicorn app.main:app --host 0.0.0.0 --port $PORT` is the actual service entrypoint (`render.yaml:6-7`).
- Root/frontend serving: the current service also serves the static frontend from `frontend/` through `app.mount("/", StaticFiles(...), name="frontend")` in `app/main.py:1354-1357`.
- Health endpoints:
  - `GET /` returns the frontend index when present, else JSON (`app/main.py:116-120`)
  - `HEAD /` returns `200` (`app/main.py:123-126`)
  - `GET /health` returns `{"ok": true}` (`app/main.py:133-135`)
- Vertex startup:
  - `init_vertex()` is invoked during module import (`app/main.py:141-168`)
  - `GCP_KEY_B64` writes a local credential file and sets `GOOGLE_APPLICATION_CREDENTIALS` (`app/services/vertex_imagen.py:18-32`)
  - `VertexImagen3()` is then constructed (`app/main.py:147`)
- Storage/R2 path:
  - Upload/presign path is implemented at `/api/r2/presign-put` (`app/main.py:873-902`)
  - Storage client supports both `R2_*` and `S3_*` names (`app/services/r2_client.py:33-37`, `71`, `97`, `116`, `131`)
- Poster API success path:
  - `generate_poster()` returns `GeneratePosterResponse` with `poster_url`, `poster_key`, optional slot assets, and warnings (`app/main.py:999-1224`)
  - Generated bytes are persisted through `store_image_and_url()` (`app/services/storage_bridge.py`)

## What is still degraded

- Kitposter runs can succeed while still being degraded.
- The current observed log state `generate_model=imagen-3.0-generate-001`, `edit_model=None`, `enabled=False` is consistent with code in `app/services/vertex_imagen3.py:108-114`.
- For kitposter modes (`kitposter1_a`, `kitposter1_b`), the service forces the Vertex branch into edit mode via `_generate_poster_with_vertex(..., force_edit=True)` (`app/services/glibatree.py:1937-1947`).
- `VertexImagen3.edit_bytes()` immediately raises when `self.enable_edit` is false or `_edit_model` is `None` (`app/services/vertex_imagen3.py:245-248`).
- That exception is caught in `generate_poster_asset()` and converted into:
  - `warnings.append("vertex_edit_failed_fallback")`
  - `degraded = True`
  - `degraded_reason = "edit_model_not_enabled"` when the error text matches the disabled-edit branch
  (`app/services/glibatree.py:1962-1971`)
- Because `primary` remains `None`, kitposter then hard-falls back to a filled locked-frame render:
  - `warnings.append("kitposter1_locked_frame_fallback")`
  - `primary = _poster_image_from_pillow(mock_frame, ...)`
  (`app/services/glibatree.py:1977-1997`)

## Root causes ranked by severity

### P0

None proven from code. Deployment is already live and healthy enough to serve traffic, produce a response, and upload artifacts.

### P1

1. Edit mode is disabled by configuration gate, so the intended kitposter edit path cannot run.
   - Evidence: `VERTEX_IMAGEN_ENABLE_EDIT` is the gating flag (`app/services/vertex_imagen3.py:113`).
   - Effect: `_edit_model` is never initialized when the flag is unset/false (`app/services/vertex_imagen3.py:130-137`).
   - Runtime consequence: kitposter requests fall into `vertex_edit_failed_fallback` then `kitposter1_locked_frame_fallback`.

2. The repository has split/ambiguous Vertex env naming across two code paths.
   - Main poster path uses `GCP_PROJECT_ID`, `GCP_LOCATION`, `VERTEX_IMAGEN_MODEL_GENERATE`, `VERTEX_IMAGEN_ENABLE_EDIT`, `VERTEX_IMAGEN_MODEL_EDIT` (`app/services/vertex_imagen3.py`).
   - The `/api/imagen/generate` factory path uses `VERTEX_PROJECT_ID`, `VERTEX_LOCATION`, `VERTEX_IMAGEN_MODEL` (`app/services/image_provider/vertex_provider.py:26-29`).
   - Effect: operators can configure the service “correctly” for one path and still leave the other path reading different names.

3. The current kitposter warning `scenario_fallback_used` is added unconditionally whenever a draft exists.
   - Evidence: `_resolve_assets_for_draft()` only checks `draft is not None`, then appends `scenario_fallback_used` (`app/services/glibatree.py:1513-1517`).
   - Effect: the warning currently does not prove a real scenario fallback happened; it is noisy/over-broad.

### P2

1. `README.md` does not match the live code truth for some Vertex env names.
   - Example: docs mention `VERTEX_PROJECT_ID`, `VERTEX_LOCATION`, `VERTEX_IMAGEN_MODEL` (`README.md:140-143`), while the main poster path reads `GCP_PROJECT_ID`, `GCP_LOCATION`, `VERTEX_IMAGEN_MODEL_GENERATE` (`app/services/vertex_imagen3.py:106-108`).

2. `config.py` contains duplicate `_parse_allowed_origins()` definitions (`app/config.py:48-64` and `147-170`).
   - This is not breaking current startup, but it is a maintenance smell in the service configuration truth source.

3. `render.yaml` still advertises `GOOGLE_APPLICATION_CREDENTIALS=/etc/secrets/service-account.json` (`render.yaml:15-16`), which is not the path currently proven healthy by runtime context.

## Why deployment is healthy but poster quality is degraded

The deployment plane and generation-quality plane are separate in the current code.

Healthy deployment means:

- process starts,
- routes mount,
- health endpoint responds,
- Vertex auth initializes,
- storage client can upload output bytes.

But poster quality depends on a deeper branch:

1. `generate_poster()` routes kitposter modes into `run_kitposter_state_machine()` (`app/main.py:1082-1098`)
2. `run_kitposter_state_machine()` calls `generate_poster_asset()` (`app/services/glibatree.py:1572-1585`)
3. `generate_poster_asset()` forces edit mode for kitposter (`app/services/glibatree.py:1937-1947`)
4. `VertexImagen3.edit_bytes()` rejects the call because edit is disabled (`app/services/vertex_imagen3.py:245-248`)
5. The service catches that and emits degraded fallback output (`app/services/glibatree.py:1962-1997`)

So the system is healthy enough to serve and store output, but not healthy enough to execute the intended edit branch.

## Current degraded/fallback causes

### Deployment/config causes

- The code-supported edit feature flag is off unless `VERTEX_IMAGEN_ENABLE_EDIT` is truthy (`app/services/vertex_imagen3.py:113`).
- The main poster path does not read `VERTEX_PROJECT_ID` / `VERTEX_LOCATION`; it reads `GCP_PROJECT_ID` / `GCP_LOCATION` (`app/services/vertex_imagen3.py:106-107`).

### Edit-model availability causes

- `edit_model=None` is expected whenever `VERTEX_IMAGEN_ENABLE_EDIT` is false/unset (`app/services/vertex_imagen3.py:113-114`).
- This is currently the first proven reason for degradation.
- Whether the configured edit model string itself is valid cannot be proven from code alone, because the current runtime never attempts to load it while edit is disabled.

### Template/layout logic causes

- Even in fallback, the service renders a deterministic locked frame via `_render_template_frame(..., fill_background=True)` (`app/services/glibatree.py:1987-1997`).
- This preserves structure, but it is not the intended background-edit pipeline, so the visual result can look flat, duplicated, or obviously templated.
- Gallery slots are forcibly padded to 4 items using product first, then scenario (`app/services/glibatree.py:1850-1876`), which can produce repeated imagery even when generation itself is healthy.

### Data/default-field causes

- `PosterInput.agent_name` is required (`app/schemas/__init__.py:127-129`).
- In one frontend payload builder, `agent_name` falls back to `messaging.channel` when missing: `posterSource.agent_name || messaging.channel || ''` (`frontend/app.js:4709-4714`).
- Since default channel values include `"email"` (`frontend/app.js:1616`, `1638`), “email” can surface in poster text even though that is a data/defaulting problem, not a deployment problem.

### Gallery fallback duplication causes

- For non-kitposter slot generation, gallery prompts are repeated up to 4 times when only one prompt exists (`app/services/glibatree.py:1769-1780`).
- After that, the final gallery array is padded to 4 entries using product/scenario fallbacks (`app/services/glibatree.py:1850-1876`).
- Both behaviors are valid defensively, but they can reduce perceived output quality.

## `/healthz` audit

Current code truth:

- `/health` exists (`app/main.py:133-135`)
- `/healthz` does not exist
- frontend helpers already expect Render to use `/health` (`frontend/app.js:1111-1114`, `README.md:50`, `218`)

Conclusion:

- Render config alone is sufficient if the health path is explicitly `/health`.
- Adding `/healthz` is not required for correctness if Render is already configured correctly now.
- Adding a `/healthz` alias would be a low-risk compatibility hardening step only, useful if old probes or external monitors still hit `/healthz`.

## Do not change yet

- Do not refactor the poster architecture or merge the two Vertex provider implementations during recovery.
- Do not rename env vars in a breaking way yet; compatibility aliases are still covering real paths.
- Do not broaden fallback behavior removal until edit-mode availability is proven with production credentials.
- Do not change gallery padding rules yet; they affect user-facing output semantics and should be tuned only after the edit path is restored.

## Recommended fix order

### P0

- None proven.

### P1

1. Confirm and enable the intended edit branch by setting `VERTEX_IMAGEN_ENABLE_EDIT=1`.
2. Validate which edit model name is actually supported in the current Vertex SDK/runtime:
   - current configured/default candidate in code: `VERTEX_IMAGEN_MODEL_EDIT` / `VERTEX_IMAGEN_EDIT_MODEL`
   - runtime proof is still missing because the branch is disabled before model load is attempted
3. Add targeted logging for edit enablement and selected env resolution so operators can distinguish:
   - feature disabled
   - model load failure
   - edit API incompatibility

### P2

1. Add optional `/healthz` alias for compatibility with stale Render or external probes.
2. Normalize env documentation and mark non-canonical aliases clearly.
3. Tighten warning semantics so `scenario_fallback_used` only appears when a real fallback occurred.
4. Review frontend draft defaulting that can set `agent_name` from `channel`.
